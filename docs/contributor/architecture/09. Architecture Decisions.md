Design Decisions
================

The following architecture decision records (ADRs) document the most significant design choices made during KIM's development. Full ADR texts are located in `docs/adr/`.

| ADR | Title                               | Status   | Summary                                                                                                                                                                                                                     |
|-----|-------------------------------------|----------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 001 | Provisioning Architecture and API   | Accepted | Defines the contract between KEB and KIM. KEB creates `Runtime` CRs with user-configurable infrastructure properties; KIM applies provider defaults and hardcoded configuration. KIM is the sole entity that calls the Gardener API. |
| 002 | Registry Cache (v1)                 | Accepted | Introduces a dedicated Registry Cache Config Controller (Option 3) to separate the concern of reading SKR cache config from applying it to Gardener. The Runtime CR gains an `imageRegistryCache` field; the new controller sets it based on `RegistryCacheConfig` CRs found on SKR. |
| 003 | Registry Cache (v2)                 | Proposed | Refines the v1 approach: uses the official `RegistryCacheConfig` CRD (from the Kyma registry-cache module), adds a validating webhook on SKR, integrates the Runtime Watcher for push-based notifications, and has the Runtime Controller manage credential Secret synchronisation between SKR and Garden. |

Additional architectural decisions captured in code comments or commit history:

| Decision                                         | Location                                             | Rationale                                                                                                                           |
|--------------------------------------------------|------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------|
| FSM state function pattern                       | `internal/controller/runtime/fsm/runtime_fsm.go`    | Complex multi-step provisioning workflow with conditional branching and variable wait times is expressed more clearly as a state machine than as a linear reconcile function. |
| Separate GardenerCluster Controller for kubeconfig rotation | `internal/controller/kubeconfig/`        | Kubeconfig rotation has a different trigger cadence (time-based) and is independent of shoot reconciliation; co-locating it in the Runtime Controller would require awkward time-based requeue logic mixed with event-driven reconciliation. |
| Audit log mandatory by default                   | `cmd/main.go`, `--audit-log-mandatory` flag          | Security and compliance requirement: provisioning should not silently succeed without audit logging. The flag allows opt-out for development environments. |
| Feature flags for new capabilities               | `cmd/main.go`                                        | New capabilities (registry cache, bootstrapper, API server ACL) are introduced behind `false`-default flags to allow gradual rollout and avoid breaking existing deployments. |
| External converter configuration file            | `pkg/config/`                                        | Provider defaults (machine types, Kubernetes versions, image versions, audit log targets) change more frequently than KIM's code. Externalising them as a JSON file lets SREs update defaults without a code change or release. |

Introduction and Goals
======================

Kyma Infrastructure Manager (KIM) is a Kubernetes operator that manages the full lifecycle of Gardener-based Kubernetes clusters for the Kyma environment. It acts as the infrastructure automation layer between the Kyma Environment Broker (KEB) and the Gardener cluster provisioning system.

Requirements Overview
---------------------

KIM fulfils the following core functional requirements:

| ID  | Requirement                                                                                                    |
|-----|----------------------------------------------------------------------------------------------------------------|
| R1  | Provision Gardener Shoot clusters based on declarative `Runtime` Custom Resources created by KEB               |
| R2  | Apply provider-specific defaults (Kubernetes version, machine images, networking ranges, audit logs) to shoots |
| R3  | Configure cluster security: OIDC, ClusterRoleBindings for administrators, API server ACL                       |
| R4  | Generate and rotate dynamic kubeconfigs for secure cluster access                                              |
| R5  | Configure the `registry-cache` Gardener extension when requested by the SKR                                    |
| R6  | Install and manage runtime bootstrapper components on SKR clusters                                             |
| R7  | Handle cluster deletion, including cleanup of secrets and extension resources                                   |
| R8  | Expose cluster lifecycle state and conditions through the `Runtime` CR status                                   |

Quality Goals
-------------

| Priority | Quality Goal       | Scenario                                                                                                         |
|----------|--------------------|------------------------------------------------------------------------------------------------------------------|
| 1        | Reliability        | Cluster provisioning and deletion must converge to the desired state even in the presence of transient API errors; all state transitions must be idempotent and safe to retry. |
| 2        | Observability      | Every state transition, error, and reconciliation event must be traceable via Kubernetes conditions, Prometheus metrics, and structured log entries. |
| 3        | Maintainability    | The Finite State Machine (FSM) structure must allow adding new provisioning steps without modifying unrelated states; the API contract between KEB and KIM must remain stable across versions. |
| 4        | Security           | Kubeconfigs must be rotated before expiry; cluster credentials must not be exposed outside the KCP namespace; network filtering must be applied as specified in the Runtime CR. |
| 5        | Configurability    | Provider defaults, audit log targets, and feature flags must be externally configurable without redeploying KIM. |

Stakeholders
------------

| Role                          | Expectations                                                                                                   |
|-------------------------------|----------------------------------------------------------------------------------------------------------------|
| Kyma Environment Broker (KEB) | Creates and deletes `Runtime` CRs; observes `status.state` and conditions to determine provisioning outcome.   |
| Platform SRE / Operator       | Configures KIM feature flags, audit log targets, machine defaults, and Kubernetes version defaults.             |
| SKR Cluster User              | Receives a functional Kubernetes cluster with OIDC access, registry cache, and properly scoped admin bindings. |
| KIM Developer                 | Understands the FSM flow and state function contracts to extend or fix provisioning behaviour.                  |
| Security Team                 | Verifies that kubeconfig rotation, network filtering, and audit logging are correctly enforced.                 |

Deployment View
===============

Infrastructure Level 1
-----------------------

KIM operates across three distinct Kubernetes clusters simultaneously.

```
┌──────────────────────────────────────────────────────────────────────────────────┐
│                       Kyma Control Plane (KCP) Cluster                           │
│                                                                                  │
│  Namespace: kcp-system                                                           │
│  ┌────────────────────────────────────────────────────────────────────────────┐  │
│  │  Deployment: infrastructure-manager                                        │  │
│  │  ┌─────────────────────────────────────────────────────────────────────┐   │  │
│  │  │  Pod: infrastructure-manager (single replica with leader election)  │   │  │
│  │  │                                                                     │   │  │
│  │  │  - Runtime Controller (25 parallel workers by default)             │   │  │
│  │  │  - GardenerCluster Controller                                      │   │  │
│  │  │  - Registry Cache Config Controller (optional)                     │   │  │
│  │  │  - Config Reload Watcher (optional)                                │   │  │
│  │  │                                                                     │   │  │
│  │  │  Mounted volumes:                                                   │   │  │
│  │  │  - gardener-kubeconfig (Secret) → /gardener/kubeconfig             │   │  │
│  │  │  - converter-config (ConfigMap) → /converter/config.json           │   │  │
│  │  └─────────────────────────────────────────────────────────────────────┘   │  │
│  │                                                                             │  │
│  │  Custom Resources managed here:                                             │  │
│  │  - Runtime CRs (watch and reconcile)                                        │  │
│  │  - GardenerCluster CRs (watch and reconcile)                                │  │
│  │  - Kubeconfig Secrets (read/write)                                          │  │
│  └────────────────────────────────────────────────────────────────────────────┘  │
│                                                                                  │
│  Monitoring: Prometheus scrapes :8080/metrics                                    │
│  Health: :8081/healthz and /readyz probes                                        │
└─────────────────────────────────────────────────┬────────────────────────────────┘
                                                  │ HTTPS (Gardener kubeconfig)
                              ┌───────────────────┴──────────────────────┐
                              │                                          │
                              ▼                                          ▼
┌─────────────────────────────────────────┐   ┌────────────────────────────────────┐
│  Gardener (Garden) Cluster              │   │  SKR Clusters (one per Runtime)    │
│  Namespace: <gardener-project>          │   │                                    │
│                                         │   │  KIM creates / reads:              │
│  KIM creates / reads:                   │   │  - kyma-system namespace           │
│  - Shoot CRs                           │   │  - ClusterRoleBindings             │
│  - Kubeconfig Secrets (from Gardener)  │   │  - OIDC ConfigMaps                 │
│  - Registry credential Secrets         │   │  - RegistryCacheConfig CRs (read)  │
│  - SecretBindings (read-only)          │   │  - Bootstrapper deployments (write) │
│                                         │   │                                    │
│  Access: via gardener kubeconfig        │   │  Access: via dynamic kubeconfig    │
│  Rate limit: 5 QPS / 10 burst          │   │  generated by Gardener              │
└─────────────────────────────────────────┘   └────────────────────────────────────┘
```

**Motivation:** The multi-cluster topology is dictated by the Gardener architecture. Gardener manages Shoot clusters from a dedicated Garden project namespace. KIM bridges the declarative `Runtime` CR on KCP with Gardener's imperative Shoot API and then reaches into the provisioned SKR for post-provisioning configuration.

Infrastructure Level 2
-----------------------

### KCP Cluster — Container Configuration

| Resource            | Value / Detail                                           |
|---------------------|----------------------------------------------------------|
| Container image     | Built from `Dockerfile` in repository root; based on distroless |
| Replica count       | 1 (leader election via `kcp-system/im-leader-election` Lease) |
| Resource requests   | Configured in `config/manager/manager.yaml`             |
| RBAC                | ClusterRole and ClusterRoleBinding in `config/rbac/`; restricted to CRD groups and `kcp-system` namespace Secrets |
| Metrics             | Exposed on `:8080/metrics` (Prometheus format)          |
| Health probes       | Liveness and readiness on `:8081`                       |
| ConfigMap reload    | Converter config changes trigger in-process reload (no pod restart required) |

### Gardener Cluster — Access Pattern

KIM connects using a kubeconfig mounted from a Secret. Requests are rate-limited to avoid overwhelming the Gardener API server during bulk operations. The `--gardener-ratelimiter-qps` and `--gardener-ratelimiter-burst` flags tune these limits.

### SKR Clusters — Access Pattern

KIM creates a short-lived Kubernetes client per reconciliation cycle using the admin kubeconfig obtained from Gardener. There is no persistent connection pool to SKR clusters; connections are created on demand and discarded after use.

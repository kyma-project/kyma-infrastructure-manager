Solution Strategy
=================

The table below summarises the most important architectural decisions and the reasoning behind them.

| Decision                                  | Choice Made                                              | Rationale                                                                                                                                                                                         |
|-------------------------------------------|----------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Programming language and framework**    | Go + kubebuilder v4 / controller-runtime                 | The Kubernetes ecosystem is Go-native. `controller-runtime` provides the reconciliation loop, leader election, and metrics integration out of the box, reducing boilerplate and aligning with Gardener's own tooling. |
| **Cluster provisioning backend**          | Gardener Shoot API                                       | Gardener abstracts hyperscaler differences (AWS, Azure, GCP, OpenStack, AliCloud) behind a uniform API, avoiding vendor lock-in at the KIM level. KIM never calls hyperscaler APIs directly.         |
| **Workflow orchestration for provisioning** | Finite State Machine (FSM) over a plain reconcile loop | Cluster provisioning is a long-running, multi-step process with conditional branching (create vs. patch vs. delete, optional steps). The FSM makes each step an isolated, testable function and makes the overall flow explicit. See ADR 001. |
| **Registry cache integration**            | Dedicated Registry Cache Config Controller + Runtime CR field | A separate controller cleanly separates the concern of reading SKR configuration from the concern of applying it to Gardener. This avoids complicating the Runtime Controller's state machine and makes time-based reconciliation straightforward. See ADR 002. |
| **Kubeconfig rotation**                   | Dedicated GardenerCluster Controller with time-based reconciliation | Rotation logic (compare age against `minimal-rotation-time * expiration-time`) is independent of cluster provisioning and needs its own requeue cadence. A separate controller keeps both loops simple. |
| **State representation**                  | `Runtime.status.state` enum + `status.conditions` list   | The state enum (`Pending`, `Ready`, `Terminating`, `Failed`) gives KEB a simple signal for automation. Conditions carry the rich, per-step detail needed for observability and troubleshooting.       |
| **Feature gating**                        | CLI flags with `false` defaults for new capabilities     | New capabilities (registry cache, bootstrapper, ACL) are introduced behind flags. This allows gradual rollout and canary testing without branching the codebase.                                   |
| **Configuration management**              | External JSON converter config + CLI flags               | Provider defaults (machine types, images, networking ranges, audit log mappings) change without code changes. Externalising them as a JSON file allows SREs to update defaults without a new KIM release. |
| **Observability**                         | Prometheus metrics + structured `logr` logging + Kubernetes Events | Three complementary signals: metrics for dashboards and alerting, structured logs for debugging, and Events for `kubectl describe` visibility. All are produced by the same state transition code paths. |

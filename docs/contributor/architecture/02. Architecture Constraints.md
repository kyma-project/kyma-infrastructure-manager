Architecture Constraints
========================

Technical Constraints
---------------------

| Constraint                                  | Explanation                                                                                                                                                              |
|---------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Kubernetes Operator pattern (kubebuilder v4) | KIM is built exclusively on the `controller-runtime` operator framework. All automation must be expressed as reconciliation loops over CRDs, not as standalone services.  |
| Gardener as the only cluster provisioner    | All Shoot cluster lifecycle operations (create, update, delete) are delegated to Gardener. KIM must not directly call hyperscaler APIs.                                   |
| Multi-cluster operation                     | KIM runs on the KCP (Kyma Control Plane) cluster and manages resources on two additional clusters: the Gardener project namespace and each individual SKR (Shoot cluster). |
| Go 1.26+ / module-based build               | The codebase must compile with Go 1.26 or later. All dependencies are managed through `go.mod`; replace directives are used for Gardener forks.                           |
| Namespace isolation on KCP                  | The controller manager is restricted to the `kcp-system` namespace; `Runtime` and `GardenerCluster` CRs may only be created there.                                       |
| Supported cloud providers                   | Only `aws`, `azure`, `gcp`, `openstack`, and `alicloud` are valid values for `spec.shoot.provider.type`. Provider-specific Gardener extensions are pre-loaded at startup. |

Organisational Constraints
--------------------------

| Constraint                                   | Explanation                                                                                                                                                 |
|----------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------|
| API contract with KEB                        | The `Runtime` CR schema is the sole interface between KEB and KIM. Changes must be backward-compatible or negotiated with the KEB team before release.       |
| ADR-driven changes                           | Significant architectural changes must be documented as Architecture Decision Records in `docs/adr/` before implementation begins.                           |
| Audit log mandatory by default               | The `--audit-log-mandatory` flag defaults to `true`; clusters for which no audit log configuration exists must fail provisioning rather than proceed silently. |
| Feature flags required for new capabilities  | New capabilities (registry cache, runtime bootstrapper, API server ACL) are hidden behind feature flags and disabled by default.                             |

Conventions
-----------

| Convention                          | Explanation                                                                                                                |
|-------------------------------------|----------------------------------------------------------------------------------------------------------------------------|
| State function naming               | FSM state functions are named with the `sFn` prefix (e.g., `sFnCreateShoot`, `sFnPatchShoot`).                            |
| Condition-based status reporting    | All observable milestones must be reflected as `metav1.Condition` entries on `Runtime.status.conditions`.                 |
| Structured logging via `logr`       | All log output must use the `logr` interface; log levels and message keys must be consistent across state functions.        |
| Prometheus metrics                  | Runtime state changes must be reflected in Prometheus counters managed through the `Metrics` interface in `internal/controller/metrics/`. |
